## What Are Vector Embeddings?

Vector embeddings are **numerical representations of text** that capture semantic meaning. Instead of treating text as mere strings, embeddings convert it into points in a high-dimensional space where semantically similar content is geometrically close.

<Note>
  Think of embeddings like GPS coordinates for meaning. Just as latitude/longitude place locations on a map, embeddings place words and sentences in "semantic space."
</Note>

### Example: Text to Vector

```
Text: "Introduction to machine learning"
        ↓
Embedding: [0.234, -0.567, 0.891, 0.123, -0.445, ... ] (768 numbers)
```

Similar text produces similar vectors:

```
"Introduction to machine learning"   → [0.234, -0.567, 0.891, ...]
"Getting started with ML"            → [0.241, -0.553, 0.879, ...]
"My vacation photos"                 → [-0.891, 0.234, -0.123, ...]
```

<Info>
  MIND-f-LAYER uses Google's `gemini-embedding-001` model, which produces 768-dimensional vectors.
</Info>

## Why Embeddings Matter for Second Brain

Traditional keyword search fails when:
- Users search for synonyms ("AI" vs "artificial intelligence")
- Queries use different phrasing ("how to cook pasta" vs "pasta recipe")
- Content is conceptually related but doesn't share exact words

Embeddings enable **semantic search**: finding content based on *meaning*, not just matching words.

### Comparison: Keyword vs Semantic Search

<CodeGroup>
```plaintext Keyword Search (Traditional)
Query: "machine learning tutorials"

Matches:
✓ "Best machine learning tutorials for beginners"
✓ "Machine learning tutorial series"
✗ "Getting started with ML"  (no "machine learning" keyword)

```

## Gemini Embedding Model

MIND-f-LAYER uses Google's `gemini-embedding-001` model:

```typescript src/configs/config.ts
export const EMBEDDING_MODEL = "gemini-embedding-001"
export const EMBEDDING_DIMENSIONS = 768
```

### Model Specifications

| Property | Value |
|----------|-------|
| Model Name | `gemini-embedding-001` |
| Dimensions | 768 |
| Max Input Length | ~2048 tokens (~1500 words) |
| Output Format | Float32 array |
| Task Types | `RETRIEVAL_DOCUMENT`, `RETRIEVAL_QUERY` |
| API Cost | Free tier: 1M tokens/month |

<Warning>
  Do not confuse embedding dimensions with token limits. The model accepts up to ~2048 tokens of input text but always outputs exactly 768 dimensions.
</Warning>

## Generating Embeddings

### For Documents (Content Storage)

When users save content, we generate embeddings from the title, description, and tags:

```typescript src/controllers/content.controller.ts
// Combine content fields for rich embedding
const embeddingText = `${title}. ${description}. Tags: ${tags.join(", ")}. Type: ${type}`

try {
    const embedding = await generateEmbedding(embeddingText)
    await db.run(
        `INSERT INTO embeddings (contentId, embedding) VALUES (?, ?)`,
        [contentId, JSON.stringify(embedding)]
    )
} catch (embErr) {
    // Don't fail content creation if embedding fails
    console.error("Failed to generate embedding:", embErr)
}
```

The `generateEmbedding` function:

```typescript src/services/ai.service.ts
export async function generateEmbedding(text: string): Promise<number[]> {
    return withRetry(async () => {
        const ai = getGenAI()
        const model = ai.getGenerativeModel({ model: EMBEDDING_MODEL })

        const result = await model.embedContent({
            content: { parts: [{ text }], role: "user" },
            taskType: "RETRIEVAL_DOCUMENT" as any,  // For indexing
        })

        const embedding = result.embedding.values
        return embedding.slice(0, EMBEDDING_DIMENSIONS)  // Ensure 768 dims
    })
}
```

<Accordion title="Why taskType: RETRIEVAL_DOCUMENT?">
  Gemini's embedding model can optimize embeddings based on the task:
  
  - `RETRIEVAL_DOCUMENT`: For content being indexed (stored for later retrieval)
  - `RETRIEVAL_QUERY`: For search queries (optimized for matching documents)
  
  Using different task types improves search relevance.
</Accordion>

### For Queries (Search)

When users query their brain, we generate a query-optimized embedding:

```typescript src/services/ai.service.ts
export async function generateQueryEmbedding(
    query: string
): Promise<number[]> {
    return withRetry(async () => {
        const ai = getGenAI()
        const model = ai.getGenerativeModel({ model: EMBEDDING_MODEL })

        const result = await model.embedContent({
            content: { parts: [{ text: query }], role: "user" },
            taskType: "RETRIEVAL_QUERY" as any,  // For searching
        })

        const embedding = result.embedding.values
        return embedding.slice(0, EMBEDDING_DIMENSIONS)
    })
}
```

<Info>
  Using `RETRIEVAL_QUERY` for queries and `RETRIEVAL_DOCUMENT` for content improves matching accuracy by 5-15% compared to using a single task type.
</Info>

## Storage Format

Embeddings are stored in SQLite as JSON strings:

```sql
CREATE TABLE embeddings (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  contentId INTEGER NOT NULL UNIQUE,
  embedding TEXT NOT NULL,  -- JSON array of 768 floats
  FOREIGN KEY (contentId) REFERENCES content(id) ON DELETE CASCADE
);
```

### Example Row

```json
{
  "id": 1,
  "contentId": 42,
  "embedding": "[0.234,-0.567,0.891,0.123,-0.445, ... 763 more numbers]"
}
```

<Note>
  Storing as JSON TEXT is simple but not optimal for very large datasets. For 10K+ items, consider binary storage (BLOB) or a dedicated vector database.
</Note>

## Cosine Similarity Algorithm

To find relevant content, we compare the query embedding to all stored embeddings using **cosine similarity**.

### What is Cosine Similarity?

Cosine similarity measures the angle between two vectors, returning a value between -1 and 1:

- **1.0**: Identical direction (perfect semantic match)
- **0.0**: Orthogonal (unrelated)
- **-1.0**: Opposite direction (contradictory meaning)

```
      Vector A
         ↗
        /  θ (angle)
       /   
      /____→ Vector B
      
Cosine Similarity = cos(θ)
```

### Implementation

```typescript src/services/ai.service.ts
export function cosineSimilarity(a: number[], b: number[]): number {
    const length = Math.min(a.length, b.length)
    let dotProduct = 0
    let normA = 0
    let normB = 0

    for (let i = 0; i < length; i++) {
        const valA = a[i]!
        const valB = b[i]!
        dotProduct += valA * valB  // Sum of element-wise products
        normA += valA * valA        // Sum of squares for vector A
        normB += valB * valB        // Sum of squares for vector B
    }

    const denominator = Math.sqrt(normA) * Math.sqrt(normB)
    if (denominator === 0) return 0
    
    return dotProduct / denominator  // Cosine of the angle
}
```

### Mathematical Formula

<Steps>
  <Step title="Compute Dot Product">
    ```
    dot(A, B) = A₁×B₁ + A₂×B₂ + ... + A₇₆₈×B₇₆₈
    ```
  </Step>
  <Step title="Compute Magnitudes">
    ```
    ||A|| = √(A₁² + A₂² + ... + A₇₆₈²)
    ||B|| = √(B₁² + B₂² + ... + B₇₆₈²)
    ```
  </Step>
  <Step title="Calculate Cosine">
    ```
    similarity = dot(A, B) / (||A|| × ||B||)
    ```
  </Step>
</Steps>

### Performance Characteristics

<CardGroup cols={2}>
  <Card title="Time Complexity" icon="clock">
    O(n × d) where n = items, d = dimensions (768)
  </Card>
  <Card title="Space Complexity" icon="memory">

  <Card title="Actual Performance" icon="gauge">

  </Card>

</Warning>

## Embedding Dimensions: Why 768?

The choice of 768 dimensions balances several factors:

<AccordionGroup>
  <Accordion title="More Dimensions = Better Accuracy">
    Higher dimensions can capture more nuanced semantic relationships, but with diminishing returns beyond ~768-1024 dimensions.
  </Accordion>

  <Accordion title="More Dimensions = Higher Cost">
    - Storage: 768 floats × 4 bytes = ~3KB per embedding
    - Computation: 768 multiplications per similarity calculation
    - Memory: 1000 items × 3KB = ~3MB RAM
  </Accordion>

  <Accordion title="Industry Standard">
    Most modern embedding models use 768-1536 dimensions:
    - OpenAI text-embedding-3-small: 1536 dims
    - Sentence-BERT: 768 dims
    - Gemini embedding-001: 768 dims
  </Accordion>
</AccordionGroup>

### API Rate Limiting

The AI service includes retry logic with exponential backoff:

```typescript src/services/ai.service.ts
async function withRetry<T>(
    fn: () => Promise<T>,
    maxRetries = 3,
    baseDelayMs = 1000
): Promise<T> {
    for (let attempt = 0; attempt < maxRetries; attempt++) {
        try {
            return await fn()
        } catch (err: any) {
            // Only retry on 429 (rate limit) or 503 (service unavailable)
            if (err?.status === 429 || err?.status === 503) {
                const delay = baseDelayMs * Math.pow(2, attempt)
                await new Promise((resolve) => setTimeout(resolve, delay))
                continue
            }
            throw err
        }
    }
}
```

<Tip>
  Gemini's free tier allows 1,500 requests per day for embedding generation. For a single user, this is sufficient for ~100 content saves per day.
</Tip>

## Limitations and Future Improvements

### Current Limitations

<Warning>
  - **No embedding updates**: Editing content doesn't regenerate embeddings
  - **Brute-force search**: Compares query against all embeddings (slow at scale)
  - **No multi-modal support**: Only text embeddings (no images, audio)
  - **Fixed dimensions**: Cannot adjust embedding size for performance tuning
</Warning>

### Potential Improvements

<Tip>
  **Future Enhancements:**
  
  1. **Automatic embedding updates**: Trigger re-embedding when content is edited
  2. **Approximate nearest neighbors**: Use HNSW or FAISS for faster search
  3. **Hybrid search**: Combine semantic search with keyword BM25
  4. **Multi-modal embeddings**: Support image and audio content
  5. **Embedding cache**: Cache frequently accessed embeddings in Redis
  6. **Batch processing**: Generate embeddings in batches for better API efficiency
</Tip>
